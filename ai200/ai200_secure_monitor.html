Azure Key Vault

Architecture & Object Types

Azure Key Vault is a cloud HSM service for securely storing and controlling access to tokens, passwords, certificates, API keys, and other secrets. For AI applications it protects API keys for Azure OpenAI, database connection strings, and storage account keys.

Key Vault URI Format

Vault URI: https://{vault-name}.vault.azure.net/
Secret URI: https://{vault-name}.vault.azure.net/secrets/{name}/{version}
Omit /{version} to always get the latest enabled version.

Azure Key Vault

Access Control: RBAC vs Access Policies

Key Vault supports two authorization models. Azure RBAC (recommended) uses Azure role assignments scoped to the vault. Access Policies (legacy) are vault-specific permissions per principal.

Azure RBAC Roles (Exam Critical)

RolePermissionsUse Case
Key Vault AdministratorFull control — manage access policies, create/delete vault objectsAdmin only
Key Vault Secrets OfficerCreate, read, update, delete, list, recover, backup secretsApplication that manages secrets lifecycle
Key Vault Secrets UserRead (get) secret values onlyApplication that only reads secrets at runtime
Key Vault ReaderRead metadata (names, properties) but NOT valuesAudit/monitoring tools
Key Vault Crypto OfficerCRUD on keys, encrypt/decrypt/sign/verifyKey management apps
Key Vault Crypto UserEncrypt/decrypt/sign/verify with existing keysApps that use keys for crypto ops
⚠
RBAC vs Access Policies — Exam Trap
  • RBAC: Permissions are Azure role assignments — scope can be subscription, resource group, or vault. Managed via IAM blade. RECOMMENDED for new workloads.
  • Access Policies: Vault-level only, set per-principal with granular operation permissions. Still works but cannot be scoped more broadly.
  • You cannot mix both models on the same vault for the same principal — the vault uses one permission model (configurable in vault settings).
  • For App Service/Functions using Key Vault References, the system-assigned managed identity needs Key Vault Secrets User role (RBAC) or Get secret permission (Access Policy).

Assigning RBAC via Azure CLI

Azure CLI — assign Key Vault Secrets User role

Full CRUD Operations

Python — SecretClient CRUD with DefaultAzureCredential
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient
from datetime import datetime, timezone, timedelta

vault_url = "https://my-vault.vault.azure.net/"
credential = DefaultAzureCredential()
client = SecretClient(vault_url=vault_url, credential=credential)

# --- CREATE / SET ---
secret = client.set_secret(
    "openai-api-key",
    "sk-prod-abc123xyz",
    content_type="text/plain",
    expires_on=datetime(2027, 1, 1, tzinfo=timezone.utc)
)
print(f"Created: {secret.name} version={secret.properties.version}")

# --- READ (latest version) ---
secret = client.get_secret("openai-api-key")
print(f"Value: {secret.value}")
print(f"Version: {secret.properties.version}")

# --- READ (specific version) ---
specific = client.get_secret("openai-api-key", version="abc123versionid")

# --- LIST all secret names (not values) ---
for prop in client.list_properties_of_secrets():
    print(f"{prop.name} — enabled={prop.enabled}, expires={prop.expires_on}")

# --- LIST all versions of a secret ---
for prop in client.list_properties_of_secret_versions("openai-api-key"):
    print(f"version={prop.version}, created={prop.created_on}")

# --- UPDATE secret properties (not value — set_secret creates new version) ---
client.update_secret_properties("openai-api-key", enabled=False)

# --- DELETE (soft delete — secret is in "deleted" state, NOT gone) ---
poller = client.begin_delete_secret("openai-api-key")
deleted = poller.result()
print(f"Deleted. Purge allowed after: {deleted.scheduled_purge_date}")

# --- RECOVER deleted secret ---
recover_poller = client.begin_recover_deleted_secret("openai-api-key")
recovered = recover_poller.result()

# --- PURGE (permanent delete — only after soft-delete, requires purge permission) ---
client.purge_deleted_secret("openai-api-key")

# --- BACKUP and RESTORE ---
backup = client.backup_secret("openai-api-key")
client.restore_secret_backup(backup)
⚠
set_secret Always Creates a New Version

Calling set_secret("name", "new-value") creates a new version — it does NOT overwrite the existing value. The new version becomes the latest active version. Old versions remain accessible by their specific version ID. To update only metadata (enabled/disabled, expiry), use update_secret_properties().

C# — SecretClient with DefaultAzureCredential (full CRUD)
using Azure.Identity;
using Azure.KeyVault.Secrets; // Azure.Security.KeyVault.Secrets NuGet

var vaultUri = new Uri("https://my-vault.vault.azure.net/");
var credential = new DefaultAzureCredential();
var client = new SecretClient(vaultUri, credential);

// SET secret
KeyVaultSecret secret = await client.SetSecretAsync("openai-api-key", "sk-abc123");
Console.WriteLine($"Version: {secret.Properties.Version}");

// GET latest version
KeyVaultSecret retrieved = await client.GetSecretAsync("openai-api-key");
Console.WriteLine(retrieved.Value);

// GET specific version
KeyVaultSecret specific = await client.GetSecretAsync("openai-api-key", "version-id");

// LIST secrets (names only, not values)
await foreach (SecretProperties prop in client.GetPropertiesOfSecretsAsync())
{
    Console.WriteLine($"{prop.Name} expires={prop.ExpiresOn}");
}

// UPDATE properties only
KeyVaultSecret toUpdate = await client.GetSecretAsync("openai-api-key");
toUpdate.Properties.Enabled = false;
await client.UpdateSecretPropertiesAsync(toUpdate.Properties);

// DELETE (soft delete)
DeleteSecretOperation op = await client.StartDeleteSecretAsync("openai-api-key");
await op.WaitForCompletionAsync();

// PURGE after soft delete
await client.PurgeDeletedSecretAsync("openai-api-key");

// RECOVER deleted secret
RecoverDeletedSecretOperation rec = await client.StartRecoverDeletedSecretAsync("openai-api-key");
await rec.WaitForCompletionAsync();

Soft Delete, Versioning & Purge Protection

Secret Versioning

Every set_secret() / SetSecretAsync() call creates a new version. The vault stores the full version history. When you get a secret without specifying a version, you always get the latest enabled version.

GRAPHIC REQUIRED — Secret Versioning Timeline

Timeline showing: version v1 (created Jan 1, disabled), version v2 (created Feb 1, active), version v3 (created after rotation, active). Arrow showing "get_secret() without version → returns v3 (latest active)".

Soft Delete Behavior

StateDescriptionReversible?
ActiveNormal state, accessible via get_secret()N/A
Deleted (Soft)Marked for deletion; in "deleted vault" for 7–90 days. NOT accessible via get_secret()Yes — recover within retention period
PurgedPermanently deleted, cannot be recoveredNo
📚
Soft Delete is Enabled by Default (Azure Portal & New Vaults)
  • Soft delete retention: configurable 7–90 days (default 90 days)
  • During soft-delete retention, the name cannot be reused (secret is in "deleted secrets" list)
  • Purge Protection: When enabled, even admins cannot purge during retention. Required for regulatory compliance. Enabling purge protection is irreversible.
  • To reuse a secret name after deletion, you must either wait for retention or explicitly purge it (if purge protection is OFF)

Secret Rotation

Secret rotation is critical for security hygiene. Azure Key Vault integrates with Event Grid to automate rotation workflows. There are two Key Vault secret lifecycle events:

Key Vault Event Grid Events

Event TypeTriggerAction
Microsoft.KeyVault.SecretNearExpiry30 days before secret expiryTrigger rotation function to set new secret version
Microsoft.KeyVault.SecretExpiredSecret expiry date reachedEmergency rotation or alert
Microsoft.KeyVault.SecretNewVersionCreatedNew version setNotify downstream services to reload config

Event-Based Rotation Architecture

GRAPHIC REQUIRED — Secret Rotation Flow

Diagram: Key Vault emits SecretNearExpiry → Event Grid → Azure Function (rotation function) → calls service API to generate new credential → set_secret() creates new version → SecretNewVersionCreated event → App Configuration or app reload.

Python — Rotation Function (Event Grid Trigger)

Python — Azure Function for secret rotation (triggered by Event Grid)
import azure.functions as func
import json, secrets
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient

app = func.FunctionApp()
kv_client = SecretClient("https://my-vault.vault.azure.net/", DefaultAzureCredential())

@app.event_grid_trigger(arg_name="event")
def rotate_secret(event: func.EventGridEvent) -> None:
    data = event.get_json()
    secret_name = data["ObjectName"]

    # Generate new secret value (example: new API key)
    new_value = secrets.token_urlsafe(32)

    # In real scenario: call the external service to create new credentials
    # e.g., regenerate storage account key, rotate DB password

    # Set new version in Key Vault
    kv_client.set_secret(secret_name, new_value)
    print(f"Rotated secret: {secret_name}")

Automated Rotation Policy (Built-in)

Azure CLI — set rotation policy on a secret
# Set rotation policy: rotate every 60 days, notify 30 days before expiry
az keyvault secret set-attributes \
  --vault-name myVault \
  --name openai-api-key \
  --expires "2025-12-31T00:00:00Z"

# Create rotation policy (available for storage account keys, etc.)
az keyvault secret rotation-policy update \
  --vault-name myVault \
  --name storage-key \
  --value '{"lifetimeActions":[{"trigger":{"timeBeforeExpiry":"P30D"},"action":{"type":"Notify"}},{"trigger":{"timeAfterCreate":"P60D"},"action":{"type":"Rotate"}}],"attributes":{"expiryTime":"P90D"}}'
Azure Key Vault

Key Vault References

Key Vault References let App Service and Azure Functions pull secret values at runtime without any SDK code. The platform automatically resolves the reference and injects the secret value as an environment variable/app setting.

Reference Syntax

App Settings — Key Vault Reference formats
# Format 1: Using Secret URI (specific version)
OPENAI_API_KEY=@Microsoft.KeyVault(SecretUri=https://my-vault.vault.azure.net/secrets/openai-api-key/abc123version)

# Format 2: Using VaultName + SecretName (always latest version — RECOMMENDED)
OPENAI_API_KEY=@Microsoft.KeyVault(VaultName=my-vault;SecretName=openai-api-key)

# Format 3: With specific version via VaultName syntax
OPENAI_API_KEY=@Microsoft.KeyVault(VaultName=my-vault;SecretName=openai-api-key;SecretVersion=abc123version)

Required Setup

  1. Enable system-assigned managed identity on App Service or Function App
  2. Assign Key Vault Secrets User role to the managed identity on the vault (RBAC) OR add a Get permission in Access Policies
  3. Set the app setting value to the @Microsoft.KeyVault(...) reference string
  4. The resolved value is accessible as os.environ["OPENAI_API_KEY"] — no SDK needed

CLI Setup

Azure CLI — enable managed identity and set KV reference
# Enable system-assigned managed identity on Function App
az functionapp identity assign \
  --name myFunctionApp \
  --resource-group myRG

# Assign Key Vault Secrets User role to the managed identity
az role assignment create \
  --role "Key Vault Secrets User" \
  --assignee $(az functionapp identity show --name myFunctionApp --resource-group myRG --query principalId -o tsv) \
  --scope $(az keyvault show --name myVault --query id -o tsv)

# Set the Key Vault Reference as an app setting
az functionapp config appsettings set \
  --name myFunctionApp --resource-group myRG \
  --settings "OPENAI_API_KEY=@Microsoft.KeyVault(VaultName=myVault;SecretName=openai-api-key)"
📚
VaultName;SecretName format (no version) = Latest Active Version

Using VaultName=...;SecretName=... without SecretVersion always resolves to the latest enabled version. This is the recommended pattern for secrets that rotate — after rotation creates a new version, all apps automatically pick it up on next secret refresh (within ~24 hours or on app restart). Use the versioned form only when you need to pin to a specific version.

builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect(new Uri("https://my-appconfig.azconfig.io"),
        new DefaultAzureCredential());

    // Load Production label settings, fall back to null label for missing keys
    options
        .Select("App:*", LabelFilter.Null)  // load defaults first

        refresh.Register("App:Sentinel", "Production", refreshAll: true)
                .SetRefreshInterval(TimeSpan.FromSeconds(30))
    );
});

// Use settings

Azure App Configuration

Dynamic Refresh & Snapshots

Sentinel Key Pattern

Rather than watching every key for changes (expensive), nominate a single sentinel key (e.g., App:Sentinel) that you update whenever you want to push a config reload. The SDK polls only this key; when it changes, it reloads all registered keys.

Azure CLI — update sentinel to trigger config reload in all apps
# After updating production config, bump the sentinel to trigger reload
az appconfig kv set \
  --name my-appconfig \
  --key "App:Sentinel" \
  --label "Production" \
  --value "$(date -u +%Y%m%dT%H%M%SZ)"  # any new value triggers reload

Python — Dynamic Refresh with Provider

Python — azure-appconfiguration-provider with refresh
from azure.appconfiguration.provider import load, WatchKey
from datetime import timedelta

config = load(
    endpoint="https://my-appconfig.azconfig.io",
    credential=DefaultAzureCredential(),
    refresh_on=[WatchKey("App:Sentinel", label="Production")],
    refresh_interval=timedelta(seconds=30)
)

# In your request handler or background loop
config.refresh()  # checks sentinel; reloads all keys if sentinel changed
max_tokens = config["App:OpenAI:MaxTokens"]

Snapshots

Snapshots are immutable point-in-time captures of all key-values in a store. Use for audit trails, rollback, and consistent configuration deployments across multiple instances.

Azure CLI — create and use a snapshot
# Create snapshot of current Production config
az appconfig snapshot create \
  --name v2-config-snapshot \
  --appconfig-name my-appconfig \
  --filters '[{"key":"App:*","label":"Production"}]' \
  --retention-period 2592000  # 30 days in seconds

# Archive (soft-delete) a snapshot
az appconfig snapshot archive \
  --name v2-config-snapshot \
  --appconfig-name my-appconfig
📚
Sentinel Key Pattern — Exam Summary

The sentinel key is the recommended pattern for dynamic refresh in production. Instead of watching all config keys (which would poll each key individually), register one sentinel key. The SDK polls only the sentinel at the configured interval (default 30s). When its ETag changes (you updated its value), the SDK reloads all watched configuration. This minimizes API calls while still enabling near-real-time config updates.

OpenTelemetry

Distributed Tracing Concepts

OpenTelemetry (OTel) is the CNCF standard for observability instrumentation. It provides a vendor-neutral SDK for collecting traces, metrics, and logs. The Azure Monitor OpenTelemetry Exporter sends OTel data to Application Insights.

GRAPHIC REQUIRED — OpenTelemetry Architecture

Diagram: Application → OTel SDK (TracerProvider, MeterProvider) → Azure Monitor Exporter → Application Insights. Shows trace → span hierarchy: one trace = one request, spans = individual operations within. W3C traceparent header propagating across microservices.

Core Concepts

ConceptDescription
TraceEnd-to-end journey of a single request across all services. Identified by a 128-bit trace_id.
SpanSingle named, timed operation within a trace. Has a span_id, start/end time, status, attributes, and events.
Parent SpanA span can have a parent span, forming a tree. Root span has no parent.
SpanContextPropagated across network boundaries: trace_id (16 bytes) + span_id (8 bytes) + trace_flags
AttributesKey-value metadata on a span (e.g., db.system=cosmosdb, http.status_code=200)
EventsTime-stamped annotations on a span (e.g., "cache miss", "retry attempt 2")
StatusUNSET, OK, or ERROR. Set to ERROR on exception.
MetricsNumeric measurements aggregated over time (counters, histograms, gauges)
LogsStructured log records correlated to traces via trace/span ID

W3C TraceContext Propagation

HeaderFormatPurpose
traceparent00-{traceId}-{spanId}-{flags}Carries trace and span IDs across HTTP calls
tracestateVendor-specific key=value pairsVendor-specific correlation data

# Format: version(2)-traceId(32hex)-parentSpanId(16hex)-flags(2hex)

Manual Span Creation

Python — manual trace instrumentation with OpenTelemetry API
from opentelemetry import trace
from opentelemetry.trace import SpanKind, StatusCode
import logging

logger = logging.getLogger(__name__)
tracer = trace.get_tracer(__name__)

def call_ai_model(prompt: str, model: str) -> str:
    with tracer.start_as_current_span(
        "ai.inference",
        kind=SpanKind.CLIENT
    ) as span:
        # Set semantic attributes
        span.set_attribute("ai.model", model)
        span.set_attribute("ai.prompt_length", len(prompt))
        span.set_attribute("service.name", "inference-service")

        try:
            # Add an event (timestamped annotation)
            span.add_event("request_started", {"model": model})

            # ... call AI model ...
            result = f"AI response for {model}"
            tokens_used = 450

            span.set_attribute("ai.tokens_used", tokens_used)
            span.add_event("request_completed", {"tokens": tokens_used})
            span.set_status(StatusCode.OK)
            return result

        except Exception as e:
            span.record_exception(e)
            span.set_status(StatusCode.ERROR, str(e))
            raise

def rag_pipeline(query: str) -> str:
    with tracer.start_as_current_span("rag.pipeline") as parent_span:
        parent_span.set_attribute("query.length", len(query))

        # Child span for vector search
        with tracer.start_as_current_span("rag.vector_search") as span:
            span.set_attribute("db.system", "cosmosdb")
            span.set_attribute("db.operation", "VectorDistance")
            chunks = []  # vector search results

        # Child span for LLM call
        with tracer.start_as_current_span("rag.llm_call") as span:
            span.set_attribute("llm.model", "gpt-4")
            result = call_ai_model(query, "gpt-4")

        return result

💡
Sampling Strategies
  • AlwaysOn (1.0): All traces sampled. Full observability, high cost.
  • AlwaysOff (0.0): No traces. Use for testing only.
  • TraceIdRatioBased (0.1): Sample 10% based on trace ID hash. Consistent — same trace is always in or out.
  • ParentBased (default): Sample based on parent span decision. Ensures complete trace capture or none.

C# — manual instrumentation with ActivitySource
using System.Diagnostics;

// Define ActivitySource once per service (static or singleton)
private static readonly ActivitySource _source =
    new("MyApp.InferenceService", "1.0.0");

public async Task<string> CallAiModelAsync(string prompt, string model)
{
    using var activity = _source.StartActivity("ai.inference", ActivityKind.Client);
    activity?.SetTag("ai.model", model);
    activity?.SetTag("ai.prompt_length", prompt.Length);

    try
    {
        activity?.AddEvent(new ActivityEvent("RequestStarted"));
        var result = await DoActualCallAsync(prompt, model);
        activity?.SetTag("ai.tokens_used", result.Usage.TotalTokens);
        activity?.SetStatus(ActivityStatusCode.Ok);
        return result.Content;
    }
    catch (Exception ex)
    {
        activity?.SetStatus(ActivityStatusCode.Error, ex.Message);
        activity?.RecordException(ex);
        throw;
    }
}

Filtering — where

KQL — where operator examples
// Time range filter (always include for performance)
requests

exceptions

Column Selection — project, project-away, extend

KQL — project and extend operators
// project: select and optionally rename columns
requests
| project timestamp, endpoint=name, durationMs=duration, success, resultCode

// project-away: drop specific columns, keep rest
requests
| project-away itemType, _ItemId, customDimensions

// extend: add computed columns
requests
| extend durationSec = duration / 1000.0
| extend isSlowRequest = duration > 2000
| extend tenantId = tostring(customDimensions["tenant"])

Aggregation — summarize

KQL — summarize operator
// Count requests per endpoint per minute
requests
| where timestamp > ago(1h)
| summarize count() by name, bin(timestamp, 5m)

// Multiple aggregates in one summarize
requests
| summarize
    totalCount = count(),
    failedCount = countif(success == false),
    avgDuration = avg(duration),
    p95Duration = percentile(duration, 95),
    p99Duration = percentile(duration, 99)
  by name

// Calculate error rate
requests
| where timestamp > ago(1h)
| summarize
    total = count(),
    errors = countif(success == false)
  by name
| extend errorRate = round(100.0 * errors / total, 2)
| order by errorRate desc

Joining — join

KQL — join operations
// inner join: only matching rows from both sides
requests
| where timestamp > ago(1h)
| join kind=inner (
    exceptions
    | where timestamp > ago(1h)
    | project operation_Id, exType=type, exMessage=outerMessage
) on operation_Id
| project timestamp, name, duration, exType, exMessage

// leftouter: all requests, with exceptions if any
requests
| join kind=leftouter (exceptions | project operation_Id, exType=type) on operation_Id

// Union: combine multiple tables
union requests, dependencies, exceptions
| where operation_Id == "abc123"
| order by timestamp asc

Last-Minute Tips

  1. Key Vault Secrets User (read-only) vs Secrets Officer (CRUD) — App Services and Functions using KV References only need Secrets User role.
  2. set_secret() always creates a NEW version — it never overwrites. The new version becomes the latest active. Old versions remain accessible by version ID.
  3. Soft delete is ON by default. Deleted secrets enter 7–90 day retention. Purge protection (once enabled) is irreversible and prevents all purges during retention.
  4. Key Vault References use @Microsoft.KeyVault(VaultName=...;SecretName=...) syntax. The app reads it as a normal env var — no SDK code needed.
  5. VaultName;SecretName (no version) always resolves to the latest ENABLED version — this is the right choice for secrets that rotate automatically.
  6. Event Grid emits SecretNearExpiry 30 days before expiry, NOT 7 days or 14 days. Know this number for the exam.
  7. App Configuration Data Reader role (not Owner) is all an application needs at runtime for reading config values.
  8. The sentinel key pattern: update ONE key to trigger reload of ALL watched settings. The SDK polls sentinel; when its ETag changes, all keys reload.
  9. Feature flags in App Config are stored under the prefix .appconfig.featureflag/ — they are special key-value pairs, not a separate storage type.
  10. OpenTelemetry spans form a tree: trace (root) → span → child spans. The trace_id is shared; span_id is unique per span.
  11. W3C traceparent header format: 00-{traceId32hex}-{spanId16hex}-{flags}. Flags=01 means sampled.
  12. configure_azure_monitor() must be called ONCE at app startup, BEFORE creating any HTTP clients, DB connections, or AI clients — so auto-instrumentation hooks are registered first.
  13. KQL where timestamp > ago(1h) — always add a time filter first for performance. Missing time filter on large tables = timeout.
  14. KQL summarize percentile(duration, 95) by name — the p95 pattern is extremely common in AI-200 exam KQL questions.
  15. KQL union requests, dependencies, exceptions | where operation_Id == "..." — the distributed trace correlation pattern shows all telemetry for one end-to-end request.
