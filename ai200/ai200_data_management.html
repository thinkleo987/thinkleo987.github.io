Azure Cosmos DB for NoSQL — Architecture & Core Concepts ★★★★★

What Is Cosmos DB for NoSQL?

Azure Cosmos DB for NoSQL is Microsoft's globally distributed, multi-model NoSQL database. It stores JSON documents, scales horizontally across any number of regions, and guarantees single-digit millisecond latency at any scale. For AI applications, it's the go-to store for structured documents enriched with AI metadata and embeddings.

Restaurant Analogy: Cosmos DB is like a global restaurant chain where every location has the exact same menu (data) available instantly, no matter which branch a customer walks into. The kitchen (Azure infrastructure) synchronizes all locations automatically.

Resource Hierarchy

Cosmos DB Account  (globally unique endpoint: account.documents.azure.com)
  └── Database      (logical grouping, shared or dedicated throughput)
        └── Container  (partitioned collection of JSON items)
              └── Item  (JSON document, up to 2 MB)

Partition Key Design

The partition key is the most important design decision for Cosmos DB performance. A bad partition key causes "hot partitions" — one partition receives most traffic while others sit idle.

★ Partition Key Rules
  • Choose a property with high cardinality (many distinct values)
  • Choose a property that evenly distributes reads and writes
  • Choose a property that frequently appears in your WHERE clauses
  • Do NOT use a timestamp as a partition key — creates hot partitions
  • Good examples: /userId, /tenantId, /productId
  • Bad examples: /createdAt, /status (only a few distinct values)

⚠ Cross-Partition Query Cost
Cross-partition queries fan out to ALL physical partitions and aggregate results. They consume significantly more RUs than single-partition queries. Always include the partition key in your query WHERE clause when possible.

Request Units (RUs) — Optimization ★★★★★

What Are Request Units?

A Request Unit (RU) is Cosmos DB's normalized currency for database operations. Every read, write, and query has an RU cost based on the CPU, memory, and I/O it consumes. You provision RUs (or use autoscale/serverless), and all operations draw from that budget.

Analogy: RUs are like fuel in a car. Every drive (operation) burns some fuel. You fill the tank (provision throughput) ahead of time. If you run out of fuel mid-journey, you get rate-limited (429 Too Many Requests).

RU Cost Reference

OperationApproximate RU CostNotes
Point read (by id + partition key)1 RU per 1 KBCheapest operation; always use when possible
Write / Upsert~5–10 RUs per 1 KBHigher due to index updates
Delete~5 RUsIndex update cost
Simple query (single partition)2–5 RUsVaries by complexity and result size
Cross-partition query10–100+ RUsFans out; avoid if possible
Query with aggregate (COUNT, SUM)5–20+ RUsMust scan matching items

Throughput Modes

ModeHow It WorksBest ForMin Cost
ProvisionedFixed RU/s you reserve; pay whether used or notPredictable, steady workloads400 RU/s
AutoscaleScales between 10%–100% of max RU/s automaticallyVariable/unpredictable workloads1000 RU/s max (scales to 100)
ServerlessPay per RU consumed, no pre-provisioningSporadic / dev workloads$0 when idle
★ RU Optimization Strategies
  1. Use point reads (ReadItemAsync) over queries — 1 RU vs 5+ RUs
  2. Include partition key in all queries — avoids cross-partition fan-out
  3. Exclude unused paths from indexing — every indexed path increases write RU cost
  4. Use projections (SELECT c.id, c.name) instead of SELECT *
  5. Batch writes — use TransactionalBatch for atomic multi-item writes
  6. Enable bulk execution mode in SDK for large bulk imports

Cosmos DB — Indexing Policies ★★★★★

Default Indexing Policy

By default, Cosmos DB automatically indexes all properties of every item. This makes queries fast but increases write RU costs (each indexed path adds work on every write). For AI workloads with large embedding arrays, this default behavior is costly — you should exclude embedding paths from indexing.

Indexing Policy JSON

{
  "indexingMode": "consistent",      // consistent | lazy | none
  "automatic": true,
  "includedPaths": [
    { "path": "/*" }               // index everything by default
  ],
  "excludedPaths": [
    { "path": "/embedding/*" },     // EXCLUDE embedding array — saves write RUs
    { "path": "/rawContent/*" },    // exclude large text fields
    { "path": "/\"_etag\"/?" }      // always exclude _etag
  ],
  "compositeIndexes": [
    [
      { "path": "/userId", "order": "ascending" },
      { "path": "/timestamp", "order": "descending" }
    ]
  ]
}
json

Indexing Modes

ModeBehaviorUse Case
consistentIndex updated synchronously on every writeDefault; queries always see latest data
lazyIndex updated asynchronously in backgroundWrite-heavy workloads; queries may miss recent items
noneNo indexing; only point reads workAppend-only logs; bulk import scenarios
✅ AI Workload Best Practice
For AI applications that store embeddings: exclude the embedding array path from indexing (/embedding/*). Embeddings are 1536+ floats — indexing them adds significant write cost and the index won't be used for vector queries anyway (vector queries use a separate vector index).

Composite Indexes

Composite indexes are required for ORDER BY on multiple fields and for queries that filter on one field and sort on another.

-- Requires a composite index on (userId ASC, timestamp DESC)
SELECT * FROM c
WHERE c.userId = 'user-123'
ORDER BY c.userId ASC, c.timestamp DESC
sql
⚠ Exam Trap — ORDER BY Multi-Field
If a query uses ORDER BY on multiple fields, Cosmos DB requires a composite index for those fields in the same order and direction. Without it, the query fails with an error — it doesn't fall back to a slower scan.

Cosmos DB — Consistency Levels ★★★★★

The Five Consistency Levels

LevelGuaranteeLatencyRU CostBest For
StrongAlways reads the latest committed write; linearizableHighest2x readsFinancial data, critical consistency
Bounded StalenessReads lag at most K versions or T seconds behind writesHigh2x readsNear-real-time apps needing bounded lag
SessionConsistent within a single client session (default)MediumNormalUser-specific reads (default & recommended)
Consistent PrefixNo out-of-order reads; never sees partial writesLowNormalSocial media feeds, ordered event logs
EventualHighest availability; reads may be staleLowestNormalNon-critical reads, recommendation engines
★ Consistency Level Memory Trick
"Strong → Bounded → Session → Prefix → Eventual"
Moving right: lower latency, lower cost, weaker consistency.
Moving left: stronger guarantees, higher cost, higher latency.

Session is the default and the most commonly used. It gives you read-your-own-writes consistency within a client session — exactly what most user-facing apps need.

Setting Consistency in SDK

    item="doc-001",
    partition_key="user-123",
    headers={"x-ms-consistency-level": "Strong"}

python

    ConsistencyLevel = ConsistencyLevel.Strong

Consistency Level Rules
You can weaken the account-level consistency for individual requests (e.g., account set to Session, request overrides to Eventual). You can NOT strengthen beyond the account-level setting per request. Strong and Bounded Staleness double the RU cost for reads.

Cosmos DB — Vector Similarity Search ★★★★★

Why Vector Search in Cosmos DB?

Cosmos DB natively supports storing embeddings and running vector similarity search using the DiskANN algorithm — a high-performance approximate nearest-neighbor index. This allows you to implement semantic retrieval and RAG directly in Cosmos DB without a separate vector database.

Step 1 — Configure Vector Embedding Policy

# Define the embedding policy when creating the container
from azure.cosmos import CosmosClient, PartitionKey
from azure.cosmos.models import VectorEmbeddingPolicy, VectorEmbedding, VectorIndexingPolicy, VectorIndex

embedding_policy = {
    "vectorEmbeddings": [
        {
            "path": "/embedding",          # JSON path to the vector field
            "dataType": "float32",         # float32 | int8 | uint8
            "distanceFunction": "cosine",   # cosine | dotproduct | euclidean
            "dimensions": 1536              # must match your embedding model output
        }
    ]
}

indexing_policy = {
    "automatic": true,
    "indexingMode": "consistent",
    "includedPaths": [{"path": "/*"}],
    "excludedPaths": [{"path": "/embedding/*"}],   # exclude from B-tree index
    "vectorIndexes": [
        {
            "path": "/embedding",
            "type": "diskANN"              # DiskANN = high-performance ANN index
        }
    ]
}

container = db.create_container_if_not_exists(
    id="documents",
    partition_key=PartitionKey(path="/userId"),
    indexing_policy=indexing_policy,
    vector_embedding_policy=embedding_policy
)
python

Step 3 — Vector Similarity Search with VectorDistance()

# Get embedding for the search query
query_embedding = get_embedding("How does Cosmos DB handle global distribution?")

# Run vector similarity search using VectorDistance()
query = """
    SELECT TOP 5
        c.id, c.content, c.category,
        VectorDistance(c.embedding, @queryEmbedding) AS score
    FROM c
    WHERE c.category = @category
    ORDER BY VectorDistance(c.embedding, @queryEmbedding)
"""

params = [
    {"name": "@queryEmbedding", "value": query_embedding},
    {"name": "@category", "value": "azure-docs"}
]

results = list(container.query_items(
    query=query,
    parameters=params,
    enable_cross_partition_query=True
))

for r in results:
    print(f"Score: {r['score']:.4f} | {r['content'][:100]}")
python
★ Vector Search Key Facts
  • VectorDistance() is the Cosmos DB function for vector similarity in queries
  • Supported distance functions: cosine, dotproduct, euclidean
  • For cosine similarity: lower score = more similar (distance, not similarity)
  • The diskANN index type enables fast approximate nearest-neighbor (ANN) search
  • You MUST exclude the embedding path from the regular B-tree index and add it to vectorIndexes
  • Metadata filters (WHERE c.category = @category) work alongside vector search

Cosmos DB — Change Feed Processor ★★★★★

What Is the Change Feed?

The change feed is an ordered, persistent log of every insert and update to items in a Cosmos DB container. It enables event-driven architectures — any time a document changes, your application can react to it automatically.

Important: The change feed does NOT include deletes by default (you can enable soft-delete patterns to capture them).

Common Use Cases

  • Trigger re-embedding when document content changes
  • Invalidate Redis cache when source data updates
  • Synchronize data to secondary read stores (denormalization)
  • Trigger Azure Functions for real-time event processing
  • Build audit logs and event sourcing systems

Change Feed Processor — Architecture

Monitored Container  →  Change Feed  →  Change Feed Processor
(source data)                              (your processing code)

                                           Lease Container
                                        (tracks read position)

Python SDK — Change Feed Processor

import asyncio
from azure.cosmos.aio import CosmosClient
from azure.cosmos import PartitionKey

async def handle_changes(changes: list, context) -> None:
    """Called when new/updated items are detected."""
    for change in changes:
        print(f"Changed item: {change['id']}")
        # Example: trigger re-embedding for updated content
        if "content" in change:
            await regenerate_embedding(change)

async def run_change_feed_processor():
    client = CosmosClient(url="...", credential=DefaultAzureCredential())
    db = client.get_database_client("ai-database")

    monitored = db.get_container_client("documents")    # source
    lease = db.get_container_client("leases")           # lease container

    # Create lease container if it doesn't exist
    await db.create_container_if_not_exists(
        id="leases",
        partition_key=PartitionKey(path="/id")
    )

    processor = monitored.get_change_feed_processor(
        processor_name="embedding-processor",
        handler=handle_changes,
        lease_container=lease
    )

    await processor.start()
    print("Change feed processor running...")
    await asyncio.sleep(3600)   # run for 1 hour
    await processor.stop()

.NET SDK — Change Feed Processor

Container monitored = db.GetContainer("documents");
Container leaseContainer = db.GetContainer("leases");

ChangeFeedProcessor processor = monitored
    .GetChangeFeedProcessorBuilder<Document>(
        processorName: "embedding-processor",
        onChangesDelegate: HandleChangesAsync
    )
    .WithInstanceName("instance-1")       // unique name per running instance
    .WithLeaseContainer(leaseContainer)
    .WithStartTime(DateTime.UtcNow)        // start from now; omit for all history
    .Build();

await processor.StartAsync();

// Handler called for each batch of changes
static async Task HandleChangesAsync(
    ChangeFeedProcessorContext context,
    IReadOnlyCollection<Document> changes,
    CancellationToken ct)
{
    foreach (var doc in changes)
    {
        Console.WriteLine($"Changed: {doc.Id}");
        await RegenerateEmbeddingAsync(doc);
    }
}
csharp
★ Change Feed Key Facts
  • Change feed captures inserts and updates only (NOT deletes by default)
  • Requires a lease container to track processing position (checkpoint)
  • processorName identifies this processor; multiple instances with different instanceName share the load
  • If the processor stops and restarts, it continues from the last checkpoint (no re-processing)
  • Multiple processors with different processorName each get their own independent view of the feed

PostgreSQL — pgvector Extension ★★★★★

What Is pgvector?

pgvector is an open-source PostgreSQL extension that adds a native vector data type and vector similarity operators. It allows PostgreSQL to store and query embedding vectors directly, without a separate vector database.

Distance Operators

OperatorDistance FunctionBest ForInterpretation
<->L2 (Euclidean) distanceSpatial similarityLower = more similar
<=>Cosine distanceText embeddings (OpenAI)Lower = more similar (0 = identical)
<#>Negative inner productNormalized vectorsMore negative = more similar
★ Which Operator to Use?
For OpenAI text embeddings (text-embedding-ada-002, text-embedding-3-small): use cosine distance (<=>). These models produce normalized vectors where cosine similarity is the natural distance metric.

Inserting Embeddings — Python

import psycopg2
from pgvector.psycopg2 import register_vector
import numpy as np

conn = psycopg2.connect("...")
register_vector(conn)    # register pgvector type with psycopg2

# Get embedding from Azure OpenAI
embedding = get_embedding("Azure PostgreSQL flexible server features")

cursor = conn.cursor()
cursor.execute(

IVFFlat Index

-- Create IVFFlat index for cosine distance
-- lists = number of clusters; rule of thumb: sqrt(num_rows)
-- For 1M rows: lists = 1000
CREATE INDEX idx_chunks_embedding_ivf
ON document_chunks
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- Set probes at query time (higher = better recall, slower)
-- Default: 1 (fast but lower recall)
-- Recommendation: probes = lists / 10 for good recall
SET ivfflat.probes = 10;

SELECT id, content, embedding <=> '[0.1, 0.2, ...]'::vector AS distance
FROM document_chunks
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 10;

HNSW Index

-- Create HNSW index for cosine distance
-- m = max connections per node (default 16; higher = better recall, more memory)
-- ef_construction = candidate list size during build (default 64; higher = better quality)
CREATE INDEX idx_chunks_embedding_hnsw
ON document_chunks
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Set ef_search at query time (higher = better recall, slower)
-- Default: 40; recommendation: >= LIMIT value
SET hnsw.ef_search = 100;

⚠ Index Only Used When Below Dimension Limit
IVFFlat supports up to 2,000 dimensions. HNSW supports up to 2,000 dimensions. OpenAI's text-embedding-ada-002 produces 1536 dimensions — within the limit. If you use a model with more dimensions, the index won't be used and queries will do a full sequential scan.

Full RAG Pipeline — Python

from openai import AzureOpenAI
import psycopg2, json, numpy as np
from pgvector.psycopg2 import register_vector

openai_client = AzureOpenAI(azure_endpoint="...", api_key="...", api_version="2024-02-01")

def rag_query(user_question: str, category: str = None, top_k: int = 5) -> str:
    """Retrieval-Augmented Generation pipeline."""

    # Step 1: Embed the user's question
    q_embedding = openai_client.embeddings.create(
        input=user_question, model="text-embedding-ada-002"
    ).data[0].embedding

    # Step 2: Retrieve similar chunks from PostgreSQL
    conn = psycopg2.connect("..."); register_vector(conn)
    cursor = conn.cursor()

    filter_clause = "AND category = %s" if category else ""
    params = [np.array(q_embedding), top_k]
    if category: params.insert(1, category)

    cursor.execute(f"""
        SELECT content, 1 - (embedding <=> %s) AS similarity
        FROM document_chunks
        WHERE 1=1 {filter_clause}
        ORDER BY embedding <=> %s
        LIMIT %s
    """, [np.array(q_embedding)] + ([category] if category else []) + [np.array(q_embedding), top_k])

    chunks = cursor.fetchall()
    conn.close()

    # Step 3: Build context from retrieved chunks
    context = "\n\n".join([f"[Relevance: {row[1]:.2f}] {row[0]}" for row in chunks])

    # Step 4: Generate answer with Azure OpenAI + context
    response = openai_client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": "Answer using only the provided context."},
            {"role": "user", "content": f"Context:\n{context}\n\nQuestion: {user_question}"}
        ]
    )
    return response.choices[0].message.content

answer = rag_query("What is the default consistency level in Cosmos DB?", category="azure-docs")

PostgreSQL — Connection Optimization ★★★★★

The Connection Problem

PostgreSQL creates a new OS process per connection. At scale, thousands of connections = thousands of processes = memory exhaustion. The solution is connection pooling.

PgBouncer — Built-in Connection Pooler

Azure Database for PostgreSQL Flexible Server includes a built-in PgBouncer connection pooler. Applications connect to PgBouncer (port 6432), which maintains a smaller pool of real PostgreSQL connections.

SessionConnection held for entire client sessionApps using session-level features (temp tables)
StatementConnection returned after each statementRarely used; doesn't support multi-statement transactions

Redis — Caching Patterns, Expiration & Invalidation ★★★★★

SET / GET with Expiration

import redis, json

r = redis.Redis(host="...", port=6380, password="...", ssl=True, decode_responses=True)

# SET with TTL (Time To Live)
r.set("cache:response:abc123", json.dumps({"answer": "Azure is a cloud platform"}), ex=3600)
# ex=3600 → expire after 3600 seconds (1 hour)
# px=60000 → expire after 60000 milliseconds
# exat=1735689600 → expire at Unix timestamp

# GET cached value
cached = r.get("cache:response:abc123")
if cached:
    result = json.loads(cached)
    print("Cache HIT")
else:
    print("Cache MISS — fetch from LLM")

# Check remaining TTL
ttl = r.ttl("cache:response:abc123")   # returns seconds; -1 = no expiry; -2 = key doesn't exist
print(f"TTL: {ttl}s")

# Update TTL on an existing key
r.expire("cache:response:abc123", 7200)  # extend to 2 hours

# Remove TTL — make key permanent
r.persist("cache:response:abc123")

# Delete key immediately (invalidate)
r.delete("cache:response:abc123")
python

Cache-Aside Pattern (Lazy Loading)

import hashlib, json

def get_ai_response(prompt: str) -> str:
    """Cache-aside: check cache first, hit LLM only on miss."""

    # 1. Generate cache key from prompt hash
    cache_key = f"llm:response:{hashlib.md5(prompt.encode()).hexdigest()}"

    # 2. Check cache
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)["answer"]

    # 3. Cache MISS — call LLM
    response = openai_client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}]
    )
    answer = response.choices[0].message.content

    # 4. Store in cache with 1-hour TTL
    r.set(cache_key, json.dumps({"answer": answer}), ex=3600)

    return answer

⚠ KEYS Command — Avoid in Production
Never use KEYS * in production — it blocks the entire Redis server while scanning. Always use SCAN with a cursor for pattern-based key enumeration. This is a common exam question about Redis best practices.

Redis — Vector Indexing with RediSearch ★★★★★

RediSearch Vector Index

Azure Managed Redis includes the RediSearch module, which adds full-text search and vector similarity search capabilities. Vectors are stored as Hash fields or JSON, and a HNSW or FLAT index enables fast KNN (K-Nearest Neighbor) queries.

Step 1 — Create a Vector Index

from redis.commands.search.field import VectorField, TextField, TagField
from redis.commands.search.indexDefinition import IndexDefinition, IndexType

# Define the index schema
schema = [
    TextField("content"),
    TagField("category"),
    VectorField(
        "embedding",

            "DISTANCE_METRIC": "COSINE",  # COSINE | L2 | IP

            "EF_CONSTRUCTION": 200    # HNSW: build quality

r.ft("idx:documents").create_index(

Last-Minute Exam Tips

  1. Point reads are cheapest in Cosmos DB. ReadItemAsync with id + partition key = 1 RU/KB. Always prefer over queries when you know the ID.
  2. Session consistency is the default. Strong and Bounded Staleness double read RU costs. You can only weaken consistency per-request, never strengthen beyond account level.
  3. Composite indexes are required for multi-field ORDER BY. Without them the query fails — it doesn't fall back.
  4. Exclude embedding paths from B-tree indexing. Add them to vectorIndexes instead. Don't index large arrays in the regular index.
  5. Change feed captures inserts and updates, NOT deletes.
  6. Lease container is required for Change Feed Processor. It stores checkpoints so the processor can resume after restart.
  7. CosmosClient should be a singleton. Creating per-request wastes TCP connections and costs extra RUs.
  8. pgvector operator for OpenAI embeddings = <=> (cosine). The index must use vector_cosine_ops to match.
  9. HNSW needs more RAM than IVFFlat. Use Memory Optimized PostgreSQL tier for HNSW indexes on large datasets.
  10. PgBouncer port = 6432 (not 5432). Transaction pooling mode is the default and best for most AI workloads.
  11. Redis: use SCAN, never KEYS. KEYS blocks the entire server on large datasets.
  12. Redis vector requires FLOAT32 bytes. Use struct.pack or numpy to convert float lists before storing.
  13. Azure Managed Redis includes RediSearch. Azure Cache for Redis Standard/Premium does NOT have it by default.
  14. IVFFlat: insert data before building the index. More efficient than building on an empty table.
  15. VectorDistance() in Cosmos DB: lower = more similar (it returns a distance, not a similarity score).
