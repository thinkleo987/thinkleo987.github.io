Module 1 — Implement Container Application Hosting ★★★★★

What Is a Container?

Explain Like I'm 15: A container is like a meal kit delivery box. The restaurant sends you the exact ingredients, exact recipe, and exact proportions. You cook it at home and it tastes identical to the restaurant — every single time.

Technical Definition: A lightweight, portable, self-contained unit of software that packages application code along with its runtime dependencies, configuration, and libraries. Containers share the host operating system kernel but run in isolated user spaces.

Containers vs. Virtual Machines

AttributeVirtual MachineContainer
Boot timeMinutesSeconds
SizeGigabytesMegabytes
OSFull OS per VMShared host kernel
IsolationFull hardware isolationProcess-level isolation
PortabilityLimitedHighly portable
Density per hostLowHigh
⚠ Critical Exam Trap
Containers share the HOST OS kernel. VMs do not. This is the single most important architectural distinction and appears on every exam.

Azure Container Instances (ACI)

ACI is the most basic Azure container service — serverless, fast, billed per CPU/memory second. Use it for short-lived, one-off, or batch container workloads.

# Deploy a container to ACI

az container create \

  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --cpu 1 \
  --memory 1.5 \
  --ports 80 \
  --dns-name-label my-ai-demo-unique123
bash

✅ Best Practice
Push once to any replicated registry — ACR automatically synchronizes the image to all replicas. You do NOT push separately to each region.

Private Link (Premium)

az network private-endpoint create \
  --name pe-acr-contoso \
  --resource-group rg-containers-demo \
  --vnet-name vnet-contoso \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id $(az acr show --name contosobankacr --query id --output tsv) \
  --group-id registry \
  --connection-name pe-connection-acr

# REQUIRED: private DNS zone for ACR
az network private-dns zone create \
  --resource-group rg-containers-demo \
  --name privatelink.azurecr.io
bash
⚠ DNS Required
Private Link alone is not enough. You MUST configure the private DNS zone privatelink.azurecr.io and link it to your VNet, otherwise DNS resolution for the ACR endpoint will fail.

Quick Task

# Build and push directly in ACR — no local Docker needed
az acr build \
  --registry contosobankacr \

Automatic Task — Git Trigger

az acr task create \
  --registry contosobankacr \
  --name build-on-commit \
  --image myapp/api:{{.Run.ID}} \
  --context https://github.com/contoso/myapp.git \
  --file Dockerfile \
  --git-access-token <pat>
bash
⚠ Base Image Update Trigger Limitation
Base image update triggers only work when the base image is in the same ACR registry or in MCR (Microsoft Container Registry). They do NOT work for images in third-party private registries.

Scheduled Task

az acr task create \
  --registry contosobankacr \
  --name nightly-build \
  --image myapp/api:nightly \
  --context https://github.com/contoso/myapp.git \
  --file Dockerfile \
  --schedule "0 2 * * *"   # 2:00 AM every day
bash

Multi-Step Task YAML

# acr-task.yaml
version: v1.1.0

steps:
  - id: build-app
    build: >
      -t $Registry/myapp/api:$ID
      -f Dockerfile .

  - id: run-tests
    cmd: $Registry/myapp/api:$ID dotnet test
    when: [build-app]      # Only runs if build-app succeeds

  - id: push-app
    push:
      - $Registry/myapp/api:$ID
      - $Registry/myapp/api:latest
    when: [run-tests]      # Only runs if tests pass
yaml

Built-in Task Variables

VariableValue
$RegistryLogin server (e.g., contosobankacr.azurecr.io)
$IDUnique run ID (alphanumeric)
$DateCurrent UTC date (YYYYMMDD)
$OSlinux or windows
 
# Run multi-step task
az acr run --registry contosobankacr --file acr-task.yaml .

# View task run logs
az acr task logs --registry contosobankacr --run-id <run-id>

Deploy from ACR

# 1. Create a Linux App Service Plan
az appservice plan create \
  --name plan-contoso-linux \
  --resource-group rg-containers-demo \
  --sku P1v3 \
  --is-linux

# 2. Create the web app with a container image
az webapp create \
  --resource-group rg-containers-demo \
  --plan plan-contoso-linux \
  --name contoso-api-app \
  --deployment-container-image-name contosobankacr.azurecr.io/myapp/api:1.0.0
bash

Managed Identity for ACR Pull

# Step 1: Assign system-assigned Managed Identity
az webapp identity assign \
  --name contoso-api-app \
  --resource-group rg-containers-demo

# Step 2: Grant AcrPull role to the web app's identity
WEBAPP_IDENTITY=$(az webapp identity show \
  --name contoso-api-app \
  --resource-group rg-containers-demo \
  --query principalId --output tsv)

az role assignment create \
  --assignee $WEBAPP_IDENTITY \
  --role AcrPull \
  --scope $(az acr show --name contosobankacr --query id --output tsv)

# Step 3: Enable Managed Identity credential usage (REQUIRED!)
az resource update \
  --ids $(az webapp show --name contoso-api-app --resource-group rg-containers-demo --query id --output tsv) \
  --set properties.acrUseManagedIdentityCreds=true
bash
★ Must Know
Setting acrUseManagedIdentityCreds=true is REQUIRED. Assigning the MI role alone is not sufficient — this property must be explicitly enabled for App Service to use Managed Identity for ACR pulls.

★ Internal DNS
Apps in the same Container Apps environment communicate via http://<app-name> — just the app name, no port, no domain suffix. Apps in different environments cannot communicate internally.

Revisions & Traffic Splitting

# Enable multiple revision mode
az containerapp revision set-mode \
  --name api-service \
  --resource-group rg-containers-demo \
  --mode multiple

# Deploy new version (creates a new revision)
az containerapp update \
  --name api-service \
  --resource-group rg-containers-demo \
  --image contosobankacr.azurecr.io/myapp/api:2.0.0

# Split traffic: 80% v1 / 20% v2
az containerapp ingress traffic set \
  --name api-service \
  --resource-group rg-containers-demo \
  --revision-weight \
    api-service--v1=80 \
    api-service--v2=20
bash
⚠ Traffic Weight Rule
Traffic weights MUST sum to exactly 100%. Both revisions must be in Active state. Updating any app setting (including env vars) creates a NEW revision automatically.

Module 7 — KEDA: Event-Driven Scaling ★★★★★

How KEDA Works

Explain Like I'm 15: Normal autoscaling hires more staff when tables fill up. KEDA is smarter — it hires staff when it sees the reservation list getting long, before the tables fill.

KEDA monitors an external event source (queue depth, message count), tells Kubernetes to scale pods accordingly, and scales down to zero when the source goes quiet.

Scalers Reference

ScalerEvent SourceScale to Zero?
azure-servicebusService Bus queue / topicYes
azure-queueAzure Storage QueueYes
azure-eventhubAzure Event HubsYes
httpHTTP concurrent requestsYes
kafkaApache Kafka lagYes
cronTime scheduleYes
cpuCPU utilizationNO
memoryMemory utilizationNO
★ Must Know — Scale to Zero
CPU and Memory scalers CANNOT scale to zero. They need at least one running pod to measure metrics. All event-based scalers (Service Bus, Queue, HTTP, Kafka, etc.) CAN scale to zero.

ScaledObject YAML — Service Bus

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-processor-scaledobject
spec:
  scaleTargetRef:
    name: order-processor-deployment
  minReplicaCount: 0    # Scale to zero when queue is empty
  maxReplicaCount: 30
  pollingInterval: 10   # Check every 10 seconds
  cooldownPeriod: 300   # Wait 5 min before scaling down
  triggers:
    - type: azure-servicebus
      metadata:
        queueName: orders
        namespace: sb-contoso
        messageCount: "5"    # 1 pod per 5 messages
      authenticationRef:
        name: servicebus-auth
yaml
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: servicebus-auth
spec:
  secretTargetRef:
    - parameter: connection
      name: servicebus-secret
      key: connection-string
yaml

Cron Scaler

triggers:
  - type: cron
    metadata:
      timezone: America/New_York
      start: "0 8 * * 1-5"    # 8 AM Mon–Fri
      end: "0 18 * * 1-5"      # 6 PM Mon–Fri
      desiredReplicas: "10"
yaml

Deployment YAML

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # 1 extra pod during update
      maxUnavailable: 0     # Never go below desired count
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: contosobankacr.azurecr.io/myapp/api:1.0.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "250m"      # 0.25 vCPU
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          readinessProbe:     # Controls traffic routing
            httpGet:
              path: /health/ready
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
          livenessProbe:      # Controls container restart
            httpGet:
              path: /health/live
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10

Last-Minute Exam Tips & One Day Before Guide

10 Last-Minute Tips

  1. Read every question twice. Container hosting questions hinge on specific requirements — "scale to zero," "GPU," "existing manifests."
  2. ACR tier = Premium for enterprise features. Any question mentioning geo-replication, Private Link, Content Trust → Premium.
  3. Managed Identity beats credentials. For AKS, App Service, Container Apps connecting to ACR → Managed Identity is the answer.
  4. AcrPull ≠ Reader. This trap appears every exam. Reader cannot pull images.
  5. readiness ≠ liveness. readiness = traffic routing, liveness = container restarts.
  6. Scale to zero = minReplicaCount: 0 AND must use an event trigger (not CPU/Memory).
  7. Revisions are immutable. Updating a Container App creates a NEW revision. Old ones persist until deactivated.
  8. Traffic weights must sum to 100%.
  9. Kubernetes Secrets = Base64, NOT encrypted. For encryption → Key Vault CSI Driver.
  10. kubectl describe pod is ALWAYS the first troubleshooting step.
