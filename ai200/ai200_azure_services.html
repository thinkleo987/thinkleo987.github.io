Service Bus vs Event Grid vs Event Hubs — Know When to Use Each
  • Service Bus: Ordered, reliable, transactional messaging. At-most-once or at-least-once. Use for back-end operations, workflow orchestration, and tasks that need guaranteed delivery and ordering (AI inference queuing).
  • Event Grid: Reactive event routing based on resource state changes. Low latency, HTTP push. Use for triggering pipelines when a Blob is uploaded or a Cosmos DB document changes.
  • Event Hubs: High-throughput telemetry/log streaming (not in this module). Use for ingesting millions of events per second (telemetry, clickstream).

Azure Service Bus

Architecture & SKUs

Azure Service Bus is a fully managed enterprise message broker providing reliable message delivery, ordering, and duplicate detection. It decouples producers from consumers, enabling resilient asynchronous AI pipelines.

GRAPHIC REQUIRED — Service Bus Resource Hierarchy

Diagram: Namespace → Queues (FIFO, competing consumers) and Topics → Subscriptions (fan-out). Shows producer → namespace → queue/topic → consumer flow with DLQ sub-queue.

📚
Exam Trap: Topics are NOT available on Basic SKU

If a scenario requires pub/sub (topics + subscriptions), the answer is Standard or Premium. Basic is queue-only. Premium is required for VNET integration, private endpoints, and message sizes over 256 KB.

Core Concepts

  • Namespace: Container for all messaging resources; endpoint is {namespace}.servicebus.windows.net
  • Queue: FIFO store for messages, single consumer (competing consumers pattern)
  • Topic: Like a queue but supports multiple subscriptions (pub/sub fan-out)
  • Subscription: Virtual queue on a topic; each gets a copy of the message
  • Dead-Letter Queue (DLQ): Sub-queue for unprocessable messages; accessible as queuename/$DeadLetterQueue
  • Message Lock: Peek-Lock mode holds message in processing state; must complete or abandon within lock duration (up to 5 min, extendable)
  • Delivery Count: Increments on each abandon; message auto-dead-letters when it hits maxDeliveryCount (default 10)

Azure Service Bus

Sending Messages

Python — Send Single and Batch Messages

Python — sending messages to a queue
from azure.servicebus import ServiceBusClient, ServiceBusMessage
from datetime import datetime, timedelta, timezone

with ServiceBusClient(namespace, credential) as client:
    with client.get_queue_sender(queue_name="ai-inference-queue") as sender:

        # Single message with properties
        msg = ServiceBusMessage(
            '{"model":"gpt-4","prompt":"Summarize this document"}',
            content_type="application/json",
            subject="inference-request",
            message_id="req-001",        # idempotency key
            time_to_live=timedelta(hours=1),
            application_properties={"priority": "high", "tenant": "acme"}
        )
        sender.send_messages(msg)

        # Scheduled message (delayed delivery)
        scheduled_msg = ServiceBusMessage("scheduled payload")
        scheduled_time = datetime.now(timezone.utc) + timedelta(minutes=30)
        seq_num = sender.schedule_messages(scheduled_msg, scheduled_time)
        # Cancel: sender.cancel_scheduled_messages(seq_num)

        # Batch — efficient for multiple messages
        batch = sender.create_message_batch()
        for i in range(10):
            try:
                batch.add_message(ServiceBusMessage(f"message {i}"))
            except ValueError:
                # Batch full — send and start new batch
                sender.send_messages(batch)
                batch = sender.create_message_batch()
                batch.add_message(ServiceBusMessage(f"message {i}"))
        sender.send_messages(batch)

Azure Service Bus

Receiving & Message Settlement

Messages are received in two modes: Peek-Lock (default, recommended) and Receive-and-Delete. Settlement methods determine what happens to a message after processing.

Receive Modes Comparison

ModeBehaviorRiskUse When
PeekLock (default)Message locked for processing; must be completed/abandonedMessage redelivered if lock expiresAt-least-once delivery (most scenarios)
ReceiveAndDeleteMessage removed immediately on receiveLost if processing failsAt-most-once; acceptable loss

Settlement Methods

MethodEffectDeliveryCount
complete_message()Permanently removes message from queueN/A (gone)
abandon_message()Returns to queue for redelivery+1 (→ DLQ at maxDeliveryCount)
dead_letter_message(reason, description)Moves to DLQ immediatelyN/A
defer_message()Hides message; retrieve later by sequence numberNot incremented
renew_message_lock()Extends lock duration (use for long processing)N/A

Python — Receiver with PeekLock

Python — PeekLock receive and settlement
from azure.servicebus import ServiceBusClient, ServiceBusReceiveMode, ServiceBusSubQueue

with ServiceBusClient(namespace, credential) as client:
    with client.get_queue_receiver(
        queue_name="ai-inference-queue",
        receive_mode=ServiceBusReceiveMode.PEEK_LOCK,  # default
        max_wait_time=5  # seconds to wait if queue is empty
    ) as receiver:

        messages = receiver.receive_messages(
            max_message_count=10,
            max_wait_time=5
        )

        for msg in messages:
            try:
                body = str(msg)
                print(f"Processing: {body}")
                # ... AI inference processing ...
                receiver.complete_message(msg)  # SUCCESS — remove from queue

            except ValueError as e:
                # Bad payload — dead-letter immediately, no retry
                receiver.dead_letter_message(
                    msg,
                    reason="InvalidPayload",
                    error_description=str(e)
                )

            except Exception as e:
                # Transient error — return to queue for retry
                receiver.abandon_message(msg)

# ServiceBusProcessor — continuous background processing
async def process_message(msg):
    print(f"Got: {str(msg)}")
    await msg.complete()

async def error_handler(args):
    print(f"Error: {args.exception}")

processor = client.get_queue_receiver(queue_name="ai-queue")
# For async continuous: use ServiceBusClient async context manager
📚
Defer Pattern — Advanced Use Case

Use defer_message() when a message cannot be processed yet but will be needed later (e.g., out-of-order message that depends on a predecessor). Deferred messages are retrieved by their sequence number via receiver.receive_deferred_messages([seq_num]). Deferred messages do NOT go to DLQ automatically — you manage them explicitly.

Azure Service Bus

Topics & Subscriptions

Topics enable pub/sub fan-out. A message sent to a topic is copied to all matching subscriptions. Each subscription has its own independent delivery and DLQ. Subscriptions can filter messages using SQL filters or Correlation filters.

Filter Types

Filter TypeSyntaxPerformanceUse Case
TrueFilter(default) matches allHighReceive all messages
FalseFiltermatches noneHighDisable subscription temporarily
CorrelationFilterExact match on system propertiesHighestMatch on Label, ContentType, CorrelationId
SqlRuleFilterSQL-92 WHERE clause on system/user propertiesMediumComplex conditions on application_properties

Python — Create Topic Subscription Filters

Python — topics, subscriptions, and filters via ServiceBusAdministrationClient
from azure.servicebus.management import ServiceBusAdministrationClient
from azure.servicebus.management import SqlRuleFilter, CorrelationRuleFilter

mgmt_client = ServiceBusAdministrationClient(namespace, credential)

# Create topic
mgmt_client.create_topic("ai-events")

# Create subscription with SQL filter (match high-priority messages)
mgmt_client.create_subscription("ai-events", "high-priority-sub")
mgmt_client.create_rule(
    "ai-events", "high-priority-sub", "high-priority-rule",
    filter=SqlRuleFilter("priority = 'high' AND tenant = 'enterprise'")
)
# Remove default TrueFilter rule
mgmt_client.delete_rule("ai-events", "high-priority-sub", "$Default")

# Create subscription with Correlation filter (faster than SQL)
mgmt_client.create_subscription("ai-events", "image-sub")
mgmt_client.create_rule(
    "ai-events", "image-sub", "image-rule",
    filter=CorrelationRuleFilter(
        label="image-processing",
        content_type="application/json"
    )
)
mgmt_client.delete_rule("ai-events", "image-sub", "$Default")

# Send to topic (same API as queue)
with client.get_topic_sender(topic_name="ai-events") as sender:
    msg = ServiceBusMessage(
        '{"type":"image","url":"..."}',
        subject="image-processing",
        application_properties={"priority": "high"}
    )
    sender.send_messages(msg)

# Receive from subscription
with client.get_subscription_receiver(
    topic_name="ai-events",
    subscription_name="image-sub"
) as receiver:
    for msg in receiver.receive_messages(max_message_count=5):
        print(str(msg))
        receiver.complete_message(msg)
Azure Service Bus

Dead-Letter Queue (DLQ)

Every queue and topic subscription has a built-in DLQ. Messages land in the DLQ when: maxDeliveryCount is exceeded, TTL expires, or they are explicitly dead-lettered by the consumer.

When Messages Go to DLQ

  • maxDeliveryCount exceeded (default 10) — repeated abandon_message() calls
  • TTL expired — time_to_live elapsed while in queue
  • Explicitly dead-lettered — consumer calls dead_letter_message()
  • Filter evaluation error — subscription filter threw an exception
  • Session lock expired (session queues) — session lock not renewed

Accessing the DLQ

Python — receive from Dead-Letter Queue
from azure.servicebus import ServiceBusSubQueue

# Access queue DLQ
with client.get_queue_receiver(
    queue_name="ai-inference-queue",
    sub_queue=ServiceBusSubQueue.DEAD_LETTER
) as dlq_receiver:
    for msg in dlq_receiver.receive_messages(max_message_count=20):
        reason = msg.dead_letter_reason
        desc = msg.dead_letter_error_description
        delivery_count = msg.delivery_count

        print(f"DLQ msg: reason={reason}, deliveries={delivery_count}")
        print(f"Original body: {str(msg)}")

        # Option 1: reprocess (fix the error, complete the DLQ message)
        dlq_receiver.complete_message(msg)

        # Option 2: re-enqueue to original queue (forward)
        with client.get_queue_sender("ai-inference-queue") as sender:
            new_msg = ServiceBusMessage(str(msg))
            sender.send_messages(new_msg)
        dlq_receiver.complete_message(msg)

# Access subscription DLQ
with client.get_subscription_receiver(
    topic_name="ai-events",
    subscription_name="image-sub",
    sub_queue=ServiceBusSubQueue.DEAD_LETTER
) as sub_dlq:
    for msg in sub_dlq.receive_messages(max_message_count=10):
        print(f"Sub DLQ: {str(msg)}")
        sub_dlq.complete_message(msg)
💡
DLQ Monitoring Pattern

Set up an Azure Function (Timer trigger, every 5 minutes) to poll the DLQ, log messages to Application Insights as custom events, and alert if DLQ depth exceeds a threshold. This is the recommended pattern for AI workloads where failed inference requests must be investigated.

Azure Service Bus

Sessions, Transactions & Processor Pattern

Sessions — Strict FIFO Per Group

Sessions enable FIFO ordering within a session group. Each message with the same session_id is processed in order by a single consumer. Required for: order processing, multi-step AI pipelines where state must be maintained per user/tenant.

Python — session-enabled queue receiver
# Send with session_id
msg = ServiceBusMessage("step-1", session_id="user-123")

# Accept a specific session
with client.get_queue_receiver(
    queue_name="session-queue",
    session_id="user-123"  # or None to accept next available session
) as receiver:
    session = receiver.session
    session.set_state("processing")      # persist session state
    for msg in receiver.receive_messages(max_message_count=10):
        receiver.complete_message(msg)
    session.set_state(None)               # clear session state when done

Transactions

Python — atomic send + receive in a transaction
# Transactions: operations either all succeed or all fail
with client.get_queue_sender("output-queue") as sender:
    with client.get_queue_receiver("input-queue") as receiver:
        msgs = receiver.receive_messages(max_message_count=1)
        if msgs:
            msg = msgs[0]
            with client.get_queue_sender("output-queue").begin_transaction() as tx:
                sender.send_messages(ServiceBusMessage("result"))
                receiver.complete_message(msg)
                tx.commit()
💡
Mnemonic: Service Bus Settlement — "CAD"

Complete (success, remove), Abandon (retry, back to queue), Dead-letter (poison, to DLQ) + Defer (later, by sequence). Remember: Complete removes, Abandon retries, Dead-letter ends retries immediately.

Azure Event Grid

Architecture & Topic Types

Azure Event Grid is a fully managed event routing service using a publish/subscribe model. Events are pushed via HTTP to handlers within milliseconds. Unlike Service Bus queues, Event Grid is event-driven (no polling) and does not guarantee message ordering.

System Topic for Storage — Common AI Pattern

When a file is uploaded to Azure Blob Storage, a Microsoft.Storage.BlobCreated event fires automatically via the Storage system topic. Subscribe an Azure Function to this event to trigger document processing, image analysis, or audio transcription pipelines. This is the canonical "upload file → trigger AI processing" pattern for AI-200.

Azure Event Grid

Retry Policy, Dead-Letter & Delivery Guarantee

Retry Schedule

Event Grid uses an exponential backoff with jitter retry policy. Events are retried up to 24 hours or the subscription's configured eventTimeToLiveInMinutes, whichever is shorter.

Retry ParameterDefaultMaxConfigurable?
maxDeliveryAttempts3030Yes (per subscription)
eventTimeToLiveInMinutes1440 (24h)1440Yes (per subscription)

Dead-Letter Configuration

Events that exceed max delivery attempts are dropped by default. To capture them, configure a dead-letter destination (Azure Blob Storage container).

Azure CLI — configure dead-letter destination
# Create storage account and container for dead-lettered events
az storage container create --name deadletter --account-name mystorageacct

# Add dead-letter destination to event subscription
az eventgrid event-subscription update \
  --name my-subscription \
  --source-resource-id $topicId \
  --deadletter-endpoint "/subscriptions/{subId}/.../blobServices/default/containers/deadletter"

Non-Retryable HTTP Responses

⚠
These Response Codes Skip Retries (Events Are Dropped or Dead-Lettered Immediately)
  • 400 Bad Request
  • 413 Request Entity Too Large
  • 403 Forbidden
  • 404 Not Found (endpoint doesn't exist)
Codes like 500, 503, 408 trigger retries. Return 200–299 from your Function/webhook to acknowledge successful delivery.
📚
Webhook Validation Handshake

When you create a webhook subscription, Event Grid sends a SubscriptionValidationEvent. Your endpoint must echo back the validationCode from the event data within 5 minutes. Azure Functions with the Event Grid trigger handle this automatically. Custom webhooks must implement it manually.

Mnemonic: Event Grid vs Service Bus

💡
"Event Grid = React, Service Bus = Transact"
  • Event Grid: Something happened → react to it. Push model, lightweight, no ordering, no transactions. "A blob was uploaded."
  • Service Bus: Something must happen → ensure it does. Pull model, ordered, transactions, DLQ, sessions. "Process this AI inference request exactly once."
  • Use Event Grid → Service Bus together: Event Grid detects change and queues it into Service Bus for reliable processing.

    database_name="ai-db",

Durable Functions

Durable Functions extend Azure Functions with state management, enabling long-running workflows. The Durable Task Framework handles checkpointing and replay so your orchestration code can survive restarts.

Function Types

TypeRoleKey Rule
OrchestratorDefines workflow logic using yield/awaitMUST be deterministic — no random, no date, no I/O (use activity for that)
ActivityPerforms actual work (DB calls, AI calls, file I/O)Can do anything; called by orchestrator
ClientStarts new orchestration instances, sends events, checks statusCalled by triggers (HTTP, timer, etc.)
EntityStateful actor model — persists state, receives signalsFor counters, aggregators, user session state

Durable Patterns

PatternDescriptionAI Use Case
Function ChainingSequential steps, output of one = input of nextLoad doc → chunk → embed → store
Fan-Out / Fan-InParallel activity calls, wait for all to completeEmbed 100 chunks in parallel
Async HTTP APIKick off long operation, poll for statusStart model training, return 202 + status URL
MonitorPoll external resource until condition metWait for batch job completion
Human InteractionPause for external approval event, resume on timeoutContent moderation approval workflow

Python v2 — Fan-Out/Fan-In Pattern

Python v2 — Durable Functions fan-out/fan-in
import azure.durable_functions as df
import azure.functions as func

myApp = df.DFApp(http_auth_level=func.AuthLevel.ANONYMOUS)

# Orchestrator function
@myApp.orchestration_trigger(context_parameter_name="context")
def doc_embedding_orchestrator(context: df.DurableOrchestrationContext):
    # Activity: chunk the document
    chunks = yield context.call_activity("chunk_document", context.get_input())

    # Fan-out: embed all chunks in parallel
    embed_tasks = [
        context.call_activity("embed_chunk", chunk)
        for chunk in chunks
    ]
    # Fan-in: wait for all embeddings
    embeddings = yield context.task_all(embed_tasks)

    # Store all embeddings
    yield context.call_activity("store_embeddings", embeddings)
    return "completed"

# Activity function
@myApp.activity_trigger(input_name="chunk")
def embed_chunk(chunk: str) -> list:
    # Actual embedding call — can do I/O, API calls, etc.
    return [0.1, 0.2]  # placeholder

# Client function (HTTP trigger that starts orchestration)
@myApp.route(route="process-doc", methods=["POST"])
@myApp.durable_client_input(client_name="client")
async def start_orchestration(
    req: func.HttpRequest,
    client: df.DurableOrchestrationClient
) -> func.HttpResponse:
    doc_url = req.get_json()["url"]
    instance_id = await client.start_new("doc_embedding_orchestrator", client_input=doc_url)
    return client.create_check_status_response(req, instance_id)
📚
Orchestrator MUST Be Deterministic

Orchestrator functions are replayed each time they resume. Never use: datetime.now() (use context.current_utc_datetime), random(), or direct I/O (HTTP/DB). All non-deterministic operations must be in Activity functions called via yield context.call_activity().

Last-Minute Tips

  1. Service Bus Basic SKU has NO topics/subscriptions — if the scenario requires pub/sub, answer is Standard or Premium.
  2. CorrelationFilter is faster than SqlRuleFilter — prefer it for high-throughput scenarios when matching on label/ContentType.
  3. DLQ is accessed by passing sub_queue=ServiceBusSubQueue.DEAD_LETTER to get_queue_receiver() — not a separate queue name.
  4. Event Grid retries up to 30 attempts over 24 hours with exponential backoff. HTTP 400/403/404/413 responses are NOT retried.
  5. Azure Functions CRON has 6 fields (starts with seconds). Standard Linux CRON has 5 (starts with minutes).
  6. HTTP trigger maximum timeout is 230 seconds regardless of plan — Azure Load Balancer limit.
  7. Durable Functions Orchestrators MUST be deterministic — never use datetime.now(), random(), or direct I/O. Use Activity functions for all side effects.
  8. Blob Trigger uses polling (up to 10 min delay). For real-time blob processing, use Event Grid trigger subscribed to storage system topic.
  9. For identity-based connections in Azure Functions, use double-underscore: SETTING__fullyQualifiedNamespace.
  10. fan-out/fan-in Durable pattern: use context.task_all() to await a list of parallel activity tasks simultaneously.
  11. Event Grid webhook subscriptions require a validation handshake — Functions with Event Grid trigger handle this automatically.
  12. Service Bus Sessions require all messages with the same session_id to be processed in FIFO order by one consumer at a time.
  13. CloudEvents 1.0 schema uses type (not eventType) and source (not topic) — know both schema formats for the exam.
  14. Premium Functions plan: no cold starts (pre-warmed), VNET integration, unlimited timeout. Consumption: cold starts, 10 min max.
  15. Service Bus Namespace → Queue/Topic hierarchy. Topics require Standard or Premium SKU. Dead-letter is always a sub-queue, not a separate entity.
